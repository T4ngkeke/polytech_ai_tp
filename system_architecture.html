Simplified

Lean Rate Limiting

Redis has been removed. Rate limiting is now handled via lightweight in-memory checks or direct DB reads against the Users table, perfectly adequate for a 20-student cohort without adding infrastructure overhead.

Simplified

Direct Streaming Proxy

The Inference Queue is gone. The Streaming Proxy now connects directly to the Compute Layer. This ensures lower latency for MVP testing and maps perfectly to standard LLM API patterns.

Unchanged

Compute Abstraction

The Model Endpoint currently points to Ollama. Because the Proxy handles standard streaming responses, migrating to Mistral API or another provider later requires only updating the endpoint URL and adding an Auth header.

Unchanged

Teacher Rules Injection

The core logic remains intact: The Prompt Controller safely injects strict pedagogical guidelines into the student's prompt before it reaches the model, ensuring the AI acts as a tutor, not an answer key.

Database Schema (PostgreSQL)

Table Status Core Fields Purpose
teacher_rules New id, teacher_id, student_id, rules_json, is_active Stores the dynamic 'poisoning' prompts designed by the teacher to restrict AI behavior.
usage_stats New id, user_id, date, tokens_used, request_count Tracks exact token consumption asynchronously to enforce the daily 1M API token limit.
users Modified id, role, daily_token_quota Contains the hard limit for each student (e.g., 50k tokens/day) used by the Rate Limiter.
messages Modified id, prompt_tokens, completion_tokens Logs the exact human-AI interaction and token cost, updated by the background task post-stream.
sessions Unchanged id, user_id Maintains standard chat session context.
